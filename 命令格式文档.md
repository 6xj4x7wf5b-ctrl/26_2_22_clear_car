# 清洁车UART控制协议文档

## 1. 协议概述

本协议定义了通过UART接口进行清洁车控制的JSON格式通信协议。协议采用请求-响应模式，支持双向通信。

### 1.1 通信参数
- **波特率**: 115200
- **数据位**: 8
- **停止位**: 1
- **校验位**: 无
- **流控**: 无

### 1.2 数据格式
- **编码**: UTF-8
- **格式**: JSON
- **校验**: CRC16（对整个JSON字节流进行校验）
- **数据包格式**: `[JSON字节流][CRC16(2字节,小端序)][\n(0x0A)]`

### 1.3 数据包结构

完整的数据包由以下部分组成：

```
[JSON字符串(UTF-8字节流)] + [CRC16校验值(2字节,小端序)] + [结束符\n(0x0A)]
```

**示例**：
```
发送的JSON字符串: {"msg_type_id":1,"msg_seq":1001,"msg_type":"cmd",...}
↓ 转换为UTF-8字节流
字节流: 7B 22 6D 73 67 5F 74 79 70 65 5F 69 64 22 3A 31 ...
↓ 计算CRC16（对JSON字节流计算）
CRC16值: 0xABCD (示例)
↓ 添加CRC和结束符
最终数据包: [JSON字节流] + [0xCD 0xAB] + [0x0A]
```

**字节序说明**：
- CRC16值采用**小端序（Little-Endian）**存储
- 例如：CRC16 = 0xABCD，则发送顺序为：`0xCD`（低字节）在前，`0xAB`（高字节）在后

### 1.4 CRC16校验算法

协议使用自定义CRC16算法，算法参数如下：

- **初始值**: 0xFFFF
- **多项式**: 0xA001（反向CRC16-IBM）
- **输入反转**: 是
- **输出反转**: 是

**算法实现**（C++参考代码）：

```cpp
uint16_t CRC16(vector<uint8_t> total_data, int valid_length)
{
    uint8_t CRC16Lo = 0xFF;
    uint8_t CRC16Hi = 0xFF;
    
    for (int i0 = 0; i0 < valid_length; i0++)
    {
        CRC16Lo = CRC16Lo ^ total_data[i0];
        for (int i1 = 0; i1 < 8; i1++)
        {
            uint8_t SaveHi = CRC16Hi;
            uint8_t SaveLo = CRC16Lo;
            CRC16Hi >>= 1;
            CRC16Lo >>= 1;
            if ((SaveHi & 1) == 1)
            {
                CRC16Lo |= 0x80;
            }
            if ((SaveLo & 1) == 1)
            {
                CRC16Hi ^= 0xA0;
                CRC16Lo ^= 1;
            }
        }
    }
    return (uint16_t)((CRC16Hi << 8) | CRC16Lo);
}
```

**CRC计算和验证流程**：

1. **发送端**：
   - 将JSON对象序列化为UTF-8字节流
   - 对JSON字节流调用CRC16函数计算校验值
   - 将CRC16值按小端序追加到JSON字节流后（低字节在前，高字节在后）
   - 最后追加结束符`\n`(0x0A)

2. **接收端**：
   - 从UART接收数据，直到收到`\n`(0x0A)结束符
   - 提取最后2字节作为接收到的CRC值
   - 对除最后2字节和结束符外的JSON字节流计算CRC16
   - 比较计算出的CRC值与接收到的CRC值，一致则校验通过

**注意事项**：
- CRC计算范围：仅对JSON字节流本身，不包括CRC字段和结束符
- 字节序：CRC16值按小端序发送（低字节在前，高字节在后）
- 如果CRC校验失败，应丢弃该数据包，不进行JSON解析

## 2. 协议结构

### 2.1 通用消息格式

所有消息都遵循以下基本结构（混合方案）：

```json
{
  "msg_type_id": 1,
  "msg_seq": 12345,
  "msg_type": "cmd",
  "msg_name": "move_lr",
  "timestamp": 1699000000000,
  "data": {
    // 具体数据内容
  }
}
```

**字段说明**：
- `msg_type_id`: 消息类型ID（整数枚举），用于快速路由和处理，必填
- `msg_seq`: 消息序列号（整数），用于请求-响应匹配，必填
- `msg_type`: 消息类型（字符串），"cmd"（命令）/ "ack"（应答）/ "report"（上报），必填
- `msg_name`: 消息名称（字符串），用于日志和调试，可选（开发环境建议包含，生产环境可省略）
- `timestamp`: 时间戳（毫秒级Unix时间戳），必填
- `data`: 消息数据内容（对象），必填

### 2.2 消息类型说明

- **cmd**: 上层机发送给设备的控制命令
- **ack**: 设备对命令的应答
- **report**: 设备主动上报的状态或数据

### 2.3 消息类型ID枚举表

#### 2.3.1 命令类型（CMD）

| msg_type_id | msg_name | 说明 |
|------------|---------|------|
| 0x01 | move_lr | 左右移动命令 |
| 0x02 | move_ud | 上下移动命令 |
| 0x03 | move_xy | 组合移动命令 |
| 0x04 | track_switch | 履带切换命令 |
| 0x05 | ball_lock | 球头锁定/解锁命令 |
| 0x06 | brush_control | 滚刷控制命令 |
| 0x07 | query_status | 查询设备状态 |
| 0x08 | query_pressure | 查询压力反馈 |

#### 2.3.2 应答类型（ACK）

| msg_type_id | msg_name | 说明 |
|------------|---------|------|
| 0x10 | ack_move_lr | 左右移动应答 |
| 0x11 | ack_move_ud | 上下移动应答 |
| 0x12 | ack_move_xy | 组合移动应答 |
| 0x13 | ack_track_switch | 履带切换应答 |
| 0x14 | ack_ball_lock | 球头锁定/解锁应答 |
| 0x15 | ack_brush_control | 滚刷控制应答 |
| 0x16 | ack_query_status | 查询状态应答 |
| 0x17 | ack_query_pressure | 查询压力应答 |

#### 2.3.3 上报类型（REPORT）

| msg_type_id | msg_name | 说明 |
|------------|---------|------|
| 0x20 | pressure | 压力反馈上报 |
| 0x21 | safety_edge | 安全触边告警上报 |
| 0x22 | device_status | 设备状态上报 |
| 0x23 | imu | IMU数据上报 |

### 2.4 方案说明
- **msg_type_id（整数）**：用于快速路由和处理，提高性能
- **msg_seq（整数）**：用于请求-响应匹配，保证消息唯一性
- **msg_name（字符串，可选）**：用于日志和调试，提高可读性

## 4. 上行反馈（设备 -> 上层机）

### 4.1 命令应答

设备收到命令后应发送应答消息：

```json
{
  "msg_type_id": 16,
  "msg_seq": 1001,
  "msg_type": "ack",
  "msg_name": "ack_move_lr",
  "timestamp": 1699000000050,
  "data": {
    "original_msg_type_id": 1,
    "original_msg_seq": 1001,
    "result": "success",
    "error_code": 0,
    "error_msg": ""
  }
}
```

**参数说明**:
- `original_msg_type_id`: 原始命令的消息类型ID
- `original_msg_seq`: 原始命令的消息序列号
- `result`: 执行结果，"success" 或 "failed"
- `error_code`: 错误码（成功时为0）
- `error_msg`: 错误描述（成功时为空字符串）

**错误码说明**:
- `0`: 成功
- `1`: 命令格式错误
- `2`: 参数超出范围
- `3`: 设备忙，无法执行
- `4`: 设备故障
- `5`: 安全保护触发

### 6.5 完整数据包示例（含CRC）

以下示例展示一个完整的UART数据包，包括JSON字节流、CRC16校验值和结束符：

**JSON消息**：
```json
{
  "msg_type_id": 1,
  "msg_seq": 1001,
  "msg_type": "cmd",
  "msg_name": "move_lr",
  "timestamp": 1699000000000,
  "data": {
    "cmd_type": "move_lr",
    "direction": "left",
    "speed": 0.5
  }
}
```

**转换为UTF-8字节流**（示例，实际长度可能不同）：
```
7B 22 6D 73 67 5F 74 79 70 65 5F 69 64 22 3A 31 2C 22 6D 73 67 5F 73 65 71 22 3A 31 30 30 31 ...
```

**计算CRC16**（对上述JSON字节流）：
```
CRC16 = 0x3A5B (示例值)
```

**完整数据包**（十六进制）：
```
[JSON字节流] + [0x5B 0x3A] + [0x0A]
     ↑              ↑           ↑
   JSON数据      CRC16(小端序)  结束符
```

**接收端处理流程**：
1. 接收数据直到遇到`0x0A`（结束符）
2. 提取最后2字节：`0x5B 0x3A`（CRC值，小端序）
3. 提取JSON字节流（除最后3字节：CRC 2字节 + 结束符1字节）
4. 对JSON字节流计算CRC16，得到`0x3A5B`
5. 比较：计算值`0x3A5B`与接收值`0x3A5B`（小端序：`0x5B 0x3A`）一致，校验通过
6. 解析JSON并处理消息

## 7. 错误处理

### 7.1 超时处理
- 命令发送后，如果500ms内未收到应答，视为超时
- 建议重发命令，最多重试3次
- 重发时应使用相同的`msg_type_id`，但`msg_seq`应递增

### 7.2 数据校验

数据校验应按照以下顺序进行：

1. **CRC校验**（优先级最高）
   - 首先进行CRC16校验，校验失败直接丢弃数据包
   - CRC校验失败时，不应进行后续的JSON解析
   - 建议记录CRC校验失败的次数，用于诊断通信质量

2. **JSON格式校验**
   - 校验JSON格式的完整性和有效性
   - JSON解析失败时，应丢弃数据包

3. **字段完整性校验**
   - 校验必填字段是否存在（msg_type_id, msg_seq, msg_type, timestamp, data）
   - 缺少必填字段时，应返回错误应答

4. **参数范围校验**
   - 校验参数范围是否合法
   - 校验msg_type_id是否在有效枚举范围内
   - 参数超出范围时，应返回错误应答（error_code=2）

### 7.3 异常情况
- 设备故障时，应通过状态上报中的`error_code`和`error_msg`字段通知上层机
- 安全保护触发时，设备应立即停止所有运动并上报告警
- 应答消息中的`error_code`非0时，上层机应根据错误码采取相应处理

### 7.4 消息序列号管理
- `msg_seq`应由发送方维护，建议使用递增序号
- 应答消息的`msg_seq`应与对应命令的`msg_seq`相同
- 上报消息的`msg_seq`由设备独立维护，建议从2000开始

## 8. 注意事项

1. **数据包格式**：完整数据包格式为 `[JSON字节流][CRC16(2字节)][\n(0x0A)]`
2. **CRC校验**：
   - CRC16计算范围：仅对JSON字节流本身（UTF-8编码）
   - CRC16值按小端序发送（低字节在前，高字节在后）
   - 接收端应先进行CRC校验，校验失败直接丢弃数据包
   - CRC校验失败不应返回应答，避免错误传播
3. **结束符**：所有数据包必须以`\n`(0x0A)结尾，用于数据包边界识别
4. **时间戳**：使用毫秒级Unix时间戳
5. **消息序列号**：`msg_seq`应保证唯一性，建议使用递增序号
6. **速度参数**：统一使用0.0~1.0的归一化值
7. **角度单位**：角度统一使用度（°），角速度使用弧度/秒（rad/s）
8. **看门狗机制**：设备应具备看门狗机制，长时间无通信时进入安全模式
9. **可选字段**：`msg_name`字段为可选，开发环境建议包含，生产环境可省略以节省带宽
10. **消息路由**：消息解析应优先使用`msg_type_id`进行路由，`msg_name`仅用于日志和调试
